# Decisión de pivot — Construir custom → adoptar ERP (Odoo / NetSuite)

**Fecha:** 2026-04-19
**Decisor:** Oliver (Product Owner)
**Contraparte que originó el reto:** Ana (ver contexto más abajo)
**Estado:** Decisión tomada. Implementación custom congelada.

---

## 1. Contexto

Durante 2 sesiones de 2026-04-19 se planificó construir un ERP custom para Arias Group cubriendo **33 capacidades** (ver listado en §5). Se entregaron y mergearon:

- **SPEC-001** — test suite del motor de cálculo (96 tests, coverage gate ≥85%/≥90%) — MERGEADA
- **SPEC-002a** — infraestructura Postgres + Docker + Alembic (db/ skeleton) — MERGEADA
- **SPEC-002b** — esquema Postgres completo + migrador SQLite→PG (PR #4) — **CERRADO sin merge**
- **SPEC-002c** — refactor `app.py` con adapter psycopg + runbook cutover (PR #5) — **CERRADO sin merge**
- **SPEC-003** (borrador) — domain cleanup para separar stages por entidad — **ARCHIVADA**

## 2. Por qué se pivota

Reconocimiento honesto tras sumar el alcance real:

| Enfoque | Tiempo a 85% funcionalidad | Coste | Riesgo |
|---|---|---|---|
| **Custom (lo que veníamos haciendo)** | 3-5 años (con Claude + Oliver solo) | ~20k€/año sostenido | Alto: bus factor 1 |
| **Odoo Enterprise + custom vertical** | 3-6 meses | ~12-25k€ año 1, ~3-5k€/año después | Bajo |
| **NetSuite** | 6-12 meses | ~100-300k€/año + 100k impl | Medio (sobrecoste) |

La lógica vertical de Arias (contenedor yeso, pipeline Fassa 26 etapas, aranceles Caribe) **no justifica rehacer CRM, inventario, contabilidad, WMS, BI, TMS desde cero**. Justifica **añadir módulos verticales sobre un ERP ya hecho**.

Ana llevaba razón al argumentar que 42.000 empresas usan Odoo/NetSuite. El CTO (Claude) reconoció haber sido optimista al defender el camino custom — la diferenciación vertical se puede conservar sin reconstruir el resto del ERP.

## 6. Plan de ejecución del pivot (propuesto por CTO)

### Paso 1 — Evaluación (2-3 semanas)

Oliver contacta **2-3 Odoo partners certificados en España/Caribe** y pide demos específicas con el caso de uso Arias:

- Distribuidor materiales construcción
- 5-10 usuarios iniciales
- Multi-país Caribe (RD, Haití, PR, Jamaica)
- Multi-moneda (EUR, USD, DOP)
- **Caso test de demo:** "me cotiza una obra de 500 m² de placas BA13 hacia Santo Domingo y me dice cuántos contenedores 40HC + precio CIF + margen".

Si Odoo de stock no resuelve ese caso en la demo, el partner tendrá que explicar qué módulo custom haría falta y cuánto costaría.

### Paso 2 — Comparativa (1 semana)

Partners probables para shortlist (investigación a tu cargo o con mi ayuda):
- **Adhoc** (España, partner gold)
- **Moldeo** (Argentina, experiencia Latam)
- **NetSuite Direct** (para presupuesto comparativo de NetSuite, aunque probablemente fuera de rango)

Criterios de decisión:
1. **Coste año 1 total** (licencias + implementación + custom module)
2. **Tiempo a producción** (quiero facturar YA — no esperar 6 meses)
3. **Soporte Caribe** — ¿hay alguien que te atienda en tu TZ?
4. **Capacidad de integrar tu microservicio vertical** (ver paso 3)
5. **Flexibilidad multi-tenant futura** (si algún día Arias crece o se escinde)

### Paso 3 — Microservicio vertical (en paralelo, ~3-4 semanas de trabajo)

Extraer el motor de cálculo del `app.py` actual y empaquetar como microservicio standalone:

```
arias-calc-service/
├── app.py               # FastAPI con 1 endpoint: POST /quote
├── calc/
│   ├── engine.py        # compute_line, estimate_containers, compute_totals
│   ├── containers.py    # CONTAINERS dict
│   └── families.py      # FAMILY_MAP
├── tests/               # Los 96 tests de SPEC-001, adaptados
├── Dockerfile
└── requirements.txt
```

- Deploy: Render ($7/mo web service)
- Consumo: Odoo llama al endpoint al generar cotización
- Retorno: JSON con líneas, contenedor recomendado, alertas, totales

**No se hace hasta que el partner Odoo esté contratado** — innecesario antes.

### Paso 4 — Implementación Odoo (3-6 meses con partner)

Responsabilidad del partner. Hitos clave:
1. Instalación + configuración inicial (semana 1-2)
2. Import catálogo 186 SKUs desde Excel master (semana 3)
3. Configuración pipeline CRM con las 26 etapas Fassa (semana 4)
4. Multi-moneda + multi-país (semana 5-6)
5. Módulo custom "Arias Calc" que llama al microservicio (semana 7-10)
6. Training Oliver + equipo (semana 11-12)
7. Go-live con primera cotización real (semana 13+)

### Paso 5 — Retirada del custom legacy (cuando Odoo esté estable)

- Archivar `https://arias-fassa.onrender.com` o redirigir a la nueva URL Odoo
- El repo `Oligonari2810/Arias_Group` se renombra o archiva (decisión futura)

## 7. Riesgos del pivot

| Riesgo | Mitigación |
|---|---|
| El partner Odoo no entiende el caso de uso vertical → mala implementación | Exigir demo con caso real antes de firmar. Doc técnico de este repo como referencia. |
| Coste licencias Odoo crece con users (~20€/user/mes) | Empezar con 5 users, escalar cuando lo justifique la operación |
| Lock-in con Odoo | Datos son tuyos siempre (export a Postgres/CSV). Riesgo real es el coste de re-entrenar al equipo si cambias de ERP. |
| Tiempo del PO (Oliver) durante 3-6 meses de implementación | Planifica — no prometas feature X a un cliente que dependa de Odoo hasta después de go-live |
| Conocimiento "Claude como CTO" se pierde al salir del repo | Este doc + las SPECs 001-003 quedan en git forever. Partner Odoo puede leerlas. |
